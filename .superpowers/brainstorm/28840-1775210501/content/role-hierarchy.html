Design — Layer C: Portal Manager

Full platform admin — account management, platform stats, DID registry

Backend Routes
GET /api/portal/stats — platform-wide counts
GET /api/portal/authorities — list all govt agency users
POST /api/portal/authorities — create new authority account
PATCH /api/portal/authorities/:id — activate / deactivate
GET /api/portal/dids — all registered DIDs
GET /api/portal/organizations — all orgs (any status)
All routes guarded by requireRole('portal_manager')
PortalManagerDashboard.tsx — 4 Tabs
Overview — stat cards: Total Orgs · Active DIDs · VCs Issued · Pending MC Actions · Approved · Rejected
Authority Accounts — table of all government_agency users with authority_type, sub_role, status badge (active/inactive). "Create Account" button opens inline form (email, name, authority_type, sub_role, temp password). Deactivate toggle per row.
DID Registry — table of all DIDs: did_string, type (parent/sub), owner name, role, created_at. Read-only view.
Organizations — table of all org applications (all statuses). Filter by status. Clicking a row shows the same detail modal as AuthorityDashboard but read-only.
Login + Routing
• Portal Manager logs in via existing /login page (no separate portal)
• redirectByRole sends portal_manager → /portal/dashboard
• New route in App.tsx: /portal/dashboard wrapped in ProtectedRouteWrapper requiring portal_manager role
• Seed script creates one Portal Manager user on first run
